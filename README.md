# x-app

## 项目介绍
`x-app` 是一个用于 **Electron 教学与演示** 的示例项目，基于 `Electron + TypeScript + Vue 3 + Rspack` 搭建。

它通过几个相对独立的模块，集中展示了桌面应用开发中常见的能力：**多标签窗口管理、悬浮球、全屏截图面板、主进程与渲染进程协作**。这个项目更强调 **API 示范性与结构可读性**，适合课堂讲解、功能拆解和二次实验。

## 关键设计摘要

### 1. 主进程负责整体编排
`src/main.ts` 是应用入口，负责：
- 创建 `TabbedWindow` 主窗口
- 初始化 `FloatBall` 悬浮球
- 注册菜单命令（开发者工具、截图开始/停止）

这种写法很适合教学，因为应用启动流程和功能入口都比较集中，便于理解 Electron 主进程的职责。

### 2. 多标签窗口采用组合式设计
`src/packages/window/` 是本项目最有代表性的部分，使用了 `BaseWindow + View + WebContentsView` 的组合方式：
- `TabbedWindow`：窗口级协调者，负责布局和生命周期
- `TabList`：管理标签集合与当前激活项
- `Tab`：保存单个标签的数据状态（标题、URL、图标）
- `TabStripView` / `TabListView` / `TabView`：分别负责标签栏、内容区和单页视图

这套拆分把“数据模型、状态管理、视图呈现”分成了几层，类关系清楚，非常适合作为 Electron 窗口架构示例。

### 3. 用小模块演示典型桌面能力
- `src/packages/float-ball/FloatBall.ts`：演示透明、置顶、可拖拽窗口
- `src/util/mouse.ts`：把拖拽过程中的鼠标事件处理单独封装出来
- `src/packages/screenshot/Screenshot.ts`：用单例窗口管理截图面板，体现主进程窗口控制与渲染层 UI 协作

### 4. 渲染层使用 Vue 独立组织界面
每个功能模块基本都带有自己的 `renderer/` 目录，例如截图模块使用 `Vue + vue-konva` 来绘制交互界面。这种结构很适合讲解：
- **主进程**：负责系统能力和窗口调度
- **渲染进程**：负责界面与交互

## 教程：如何用这个项目学习 Electron

这一节不是泛泛介绍，而是直接对应当前仓库里的代码，适合边看文件边学习。

### 第 1 步：先看应用入口 `src/main.ts`
这里是整个项目的启动编排点，建议重点关注：
- `app.whenReady()`：Electron 应用初始化完成后的入口
- `Menu.setApplicationMenu(...)`：菜单与快捷键注册方式
- `new TabbedWindow({})`：主窗口创建
- `new FloatBall()`：悬浮球模块初始化
- `ScreenShotWindow.start()` / `stop()`：截图功能的调用入口

学完这一部分，你应该先建立一个认识：**主进程负责组织功能模块，而不是直接写页面细节。**

### 第 2 步：理解多标签窗口系统 `src/packages/window/`
这是项目里最核心、也最适合教学的部分。推荐按这个顺序读：

1. `Tab.ts`
   - 这是单个标签的数据模型
   - 保存 `id`、`url`、`title`、`icon`
   - 使用 `EventEmitter` 在属性变化时发出通知

2. `TabList.ts`
   - 负责管理多个 `Tab`
   - 维护当前激活的标签
   - 对外发出 `tab-added`、`tab-removed`、`active-tab-changed` 等事件

3. `TabbedWindow.ts`
   - 是整个标签窗口的协调者
   - 负责把 `TabList`、`TabStripView`、`TabListView` 组装到一起
   - 负责窗口布局和“新开页面时转为新标签”这类策略

4. `TabStripView.ts` / `TabListView.ts` / `TabView.ts`
   - `TabStripView`：顶部标签栏
   - `TabListView`：标签页内容区域管理器
   - `TabView`：单个网页内容视图

这一套代码非常适合讲清楚 **模型层、状态层、视图层** 的协作关系。

### 第 3 步：看悬浮球模块 `src/packages/float-ball/`
`FloatBall.ts` 演示了 Electron 中比较常见的一类窗口：
- 无边框
- 透明背景
- 始终置顶
- 可拖拽

同时它配合 `src/util/mouse.ts`，把拖拽过程中需要的鼠标事件提取成了单独工具。这是一个很好的示例：**把通用交互逻辑从窗口类中拆出来。**

### 第 4 步：看截图模块 `src/packages/screenshot/`
`Screenshot.ts` 使用了单例方式管理截图窗口：
- 已存在实例时直接聚焦
- 不存在时再创建
- 关闭后把实例重置为 `null`

这很适合用来讲“**系统型功能窗口**”的管理方式。配合它的 `renderer/` 代码，还可以继续讲主进程负责窗口、渲染层负责交互界面这一分工。

### 第 5 步：理解这个项目想教会你的重点
通过这个仓库，建议重点掌握下面 5 个概念：

1. **主进程与渲染进程分工**
2. **窗口对象与视图对象如何组合**
3. **事件驱动的数据同步方式**
4. **桌面能力（菜单、悬浮、截图）的组织方式**
5. **如何把功能按模块拆分到 `packages/` 中**

### 第 6 步：建议的动手练习
如果你要拿这个项目做教学或自学，可以直接布置这些练习：

- **练习 1：修改默认打开页面**  
  在 `src/main.ts` 中调整 `mainWindow.addNewTab(...)` 的初始地址。

- **练习 2：给标签页增加更多状态**  
  例如为 `Tab` 增加 `loading` 字段，再让 `TabStripView` 响应这个状态。

- **练习 3：扩展一个新的工具窗口**  
  参考 `FloatBall` 或 `ScreenShotWindow`，实现一个新的演示模块。

- **练习 4：补生命周期清理**  
  给 `TabStripView`、`TabListView`、`TabView` 设计统一的 `destroy()` 逻辑。

- **练习 5：做一次安全性升级**  
  把演示用途的宽松配置，逐步调整成更接近生产环境的写法。

### 第 7 步：适合课堂怎么讲
如果你要拿它做一节课，可以按这个顺序展开：

1. 先讲 Electron 的进程模型
2. 再从 `src/main.ts` 讲应用入口
3. 用 `window/` 模块讲清多标签窗口架构
4. 用 `float-ball/` 讲特殊窗口能力
5. 用 `screenshot/` 讲独立功能模块和渲染层 UI
6. 最后再讨论安全性、清理机制和工程化改进

这样讲下来，学生既能看到“能跑的桌面功能”，又能理解背后的结构设计。

## 设计评价（更新）

### 🎯 结论
如果从 **Electron 教学示例** 的角度来看，这个项目我会给出 **8.5/10**；如果从 **生产项目完备度** 来看，更接近 **6.5/10**。

也就是说：**它很适合拿来讲解 Electron 的核心能力与架构思路，但还没有完全朝生产级工程化收口。**

## ✅ 这个项目值得肯定的地方

### 1. 类关系清晰，适合教学
窗口模块的依赖链路比较清楚：

```text
TabbedWindow
 ├─ TabList
 │   └─ Tab
 ├─ TabStripView
 └─ TabListView
     └─ TabView
```

- `Tab` 负责单个标签的数据状态
- `TabList` 负责集合和激活逻辑
- `TabbedWindow` 负责调度和布局
- 各个 `View` 负责具体展示

这套关系非常容易讲清楚“模型、状态、视图协作”的基本思路。

### 2. 模块划分贴近真实 Electron 场景
项目没有把所有代码塞进一个窗口里，而是按功能拆成了：
- `window`：多标签窗口系统
- `float-ball`：悬浮球
- `screenshot`：截图能力
- `renderer/`：各模块自己的前端界面

对于教学来说，这种按功能切包的方式很直观。

### 3. 展示了比较典型的 Electron API 用法
代码中实际覆盖了不少课堂上常讲的知识点：
- `BaseWindow` / `BrowserWindow`
- `View` / `WebContentsView`
- 菜单与快捷键
- 屏幕尺寸获取与置顶窗口
- IPC 事件通信
- 主进程与渲染进程配合

所以它不是“纯概念项目”，而是一个可讲、可改、可演示的样例。

## ⚠️ 目前的设计短板

### 1. 生命周期清理还不够完整
像 `TabStripView`、`TabListView`、`TabView` 中的监听器注册较多，但销毁与解绑不够系统。在教学中这是可以接受的简化，但如果继续扩展，就要尽快补上 `destroy()` 和统一清理机制。

### 2. 主进程与渲染层边界还能更明确
当前有些逻辑还是直接由窗口类驱动具体视图行为，适合作为“先跑起来”的 demo；但若要继续演进，建议补一层更清晰的状态/服务抽象，减少直接耦合。

### 3. 安全配置偏向演示便利
多个窗口都启用了：
- `nodeIntegration: true`
- `contextIsolation: false`
- `sandbox: false`

这对教学演示很方便，但需要明确说明：**这不是生产环境的推荐默认值**。
